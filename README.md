# 🎭 Persona — role plugin for OpenCode

Persona adapts the OpenCode assistant to each user's **professional role**. The first time you open a session it asks for your role (developer, architect, analyst, or QA), stores it in [Engram](https://github.com/Gentleman-Programming/engram), and from then on automatically loads that role's instructions — together with your communication preferences and the project's conventions — at the start of every session.

Every reply generated by the plugin starts with the `🎭 Persona plugin:` marker, so you can tell at a glance that Persona answered.

```
🎭 Persona plugin: active role — Developer.
```

## Features

### Automatic role detection and loading

- **First-session onboarding**: the plugin introduces itself, asks for your role (options 1-4), and saves it when you answer. This happens only once per user and machine.
- **Automatic injection**: in every new session, the active role's instructions are injected into the assistant's context without user intervention.
- **Per-project instructions**: each project defines how each role behaves in `harness/user-roles/` (`DEV.md`, `ARQ.md`, `BA.md`, `QA.md`). The same role can behave differently in different projects.
- **Hot role switching**: just say "change my role to QA" in the chat; the plugin updates the stored role without duplicating it and without repeating the onboarding.

### Communication preferences (personal scope)

Set by simply saying so in the chat: "always reply in English", "be more brief".

- **Reply language** and **level of detail**, each field updatable separately.
- **Personal** scope: they travel with you and apply in all your sessions, in any project.

### Project working conventions (project scope)

Recorded by simply saying so in the chat: "commits in this repo are written in English".

- Up to **20 conventions per project**, no duplicates.
- **Project** scope: they are injected only in your sessions within that project.

### Status query

Ask "what do I have recorded in Persona?" and the assistant queries Engram (the `get_persona_status` tool) and answers with the actual role, preferences, and conventions you have stored, not with repository documents.

### Language behavior

The plugin's prompts are written in English, but the assistant **always replies in the language you write in**: chat in Spanish and you get Spanish replies. If you save a language preference ("always reply in English"), that saved preference takes precedence over mirroring your message language.

### Robustness

- **Graceful degradation**: if Engram is unavailable, the plugin never blocks the session; the assistant runs with its default behavior and the role is asked again once Engram is reachable. Hung connections are cut off by a timeout (8 s).
- **Independent failures**: if preferences or conventions fail to load, the role is injected anyway.
- **Subagent sessions ignored**: injection only happens in main sessions.
- **Missing role file**: if the role's `.md` file is missing from `harness/user-roles/`, the assistant says so and continues with its default behavior.
- **Diagnostics**: all activity is logged to `.opencode/persona.log` (or `~/.persona/persona.log` if the project is not writable).

## Tools exposed by the plugin

| Tool | What it does |
|------|--------------|
| `save_user_role` | Saves or updates the user's role (upsert, no duplicates). |
| `save_user_preferences` | Saves the reply language and/or level of detail. |
| `save_project_convention` | Adds a working convention to the current project. |
| `get_persona_status` | Returns the role, preferences, and conventions recorded in Engram. |

There is no need to invoke them manually: the assistant uses them when you express a role, a preference, or a convention in the chat.

## What is shared and what is not?

This is the most common question, so here is the full table:

| Data | Scope in Engram | Shared across your projects? | Shared with the team? |
|------|-----------------|------------------------------|------------------------|
| Role | Personal | ✅ Yes — asked once, applies everywhere | ❌ No |
| Preferences (language, detail) | Personal | ✅ Yes | ❌ No |
| Project conventions | Project | ❌ No — each project has its own | ❌ No |

**Conventions are NOT shared across projects**: they stay bound to the project where you recorded them. And no data is shared between people: Engram is a **local, per-user, per-machine** database, so each team member records their own role, preferences, and conventions, even when working on the same repository.

## Global or per-project installation?

The plugin is enabled **per project** (opt-in): the `plugin` entry lives in each project's `opencode.json`, so it only activates in the projects that declare it. This is a design decision, not an accidental limitation:

- **Role instructions live in each project** (`harness/user-roles/`). Without that folder, the plugin has no content to inject and degrades to the default behavior, so a global installation would add little in unprepared projects.
- A global `~/.config/opencode/opencode.json` can also declare plugins, but Persona does not recommend that channel, for the same reason.

What is effectively "global" is your **role and preferences**: they are saved once with personal scope and automatically follow you into every project that enables the plugin. Enabling per project does not mean reconfiguring per project.

## Requirements

- **OpenCode** (CLI or desktop app).
- **Engram** installed and reachable on the PATH — the plugin launches `engram mcp` as a subprocess.
- A `harness/user-roles/` folder in the project root with one instruction file per role — created during installation (step 2) by copying `templates/user-roles/`.

Node.js and npm are **not** required to use the plugin: OpenCode downloads and installs it automatically with its bundled Bun. They are only needed to contribute to this repository.

## Installation

1. Add the plugin to your project's `opencode.json` (create the file in the project root if it does not exist):

   ```json
   {
     "$schema": "https://opencode.ai/config.json",
     "plugin": ["opencode-persona"]
   }
   ```

   OpenCode installs the package automatically the next time it starts in that project.

2. Create the role instructions folder in your project by copying `templates/user-roles/` from [this repository](https://github.com/RuloGB/opencode-persona/tree/main/templates/user-roles), then adapt each file to your project:

   ```bash
   # from your project root
   cp -r <path-to-this-repo>/templates/user-roles harness/user-roles
   ```

3. Open OpenCode in the root of your project. The first session asks for your role.

See [docs/INSTALL.md](docs/INSTALL.md) for the detailed guide (version pinning, updates, local development), verification steps, and troubleshooting.

## Repository structure

```
.
├── src/
│   ├── index.ts             # Plugin entry point
│   └── *.ts                 # Internal modules (roles, preferences, conventions, Engram client...)
├── test/                    # Test suite (node:test + a fake Engram MCP server)
├── templates/
│   └── user-roles/          # Role instruction templates: DEV.md, ARQ.md, BA.md, QA.md, _TEMPLATE.md
├── docs/
│   └── INSTALL.md           # Detailed installation guide, verification, and troubleshooting
└── dist/                    # Built ESM output (generated by `npm run build`, published to npm)
```

`harness/user-roles/` is not part of this repository: each consuming project creates it (for example, by copying `templates/user-roles/`) and owns its contents.

## Tests

```bash
npm install
npm run typecheck && npm test
```

The suite does not need Engram installed: it uses a fake MCP server included in `test/helpers/` and never touches your real database.
